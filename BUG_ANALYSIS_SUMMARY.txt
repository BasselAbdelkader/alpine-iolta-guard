================================================================================
CRITICAL BUGS (MUST FIX BEFORE PRODUCTION)
================================================================================

BUG #1: BALANCE CALCULATION - VOIDED TRANSACTIONS IN RUNNING BALANCE
Location: /backend/apps/clients/api/views.py lines 137-143
Impact: Historical balance reports are INCORRECT
Issue: balance_history endpoint doesn't check transaction status when calculating
       running balance. Voided transactions could inflate balances.

BUG #5: CASE DEPOSIT CREATION - NOT ATOMIC
Location: /backend/apps/clients/models.py lines 301-305
Impact: Case balance can MISMATCH sum of transactions
Issue: Case is saved BEFORE deposit transaction is created. If deposit creation 
       fails, case exists without required deposit, causing data inconsistency.

BUG #6: CHECK NUMBER RACE CONDITION
Location: /backend/apps/bank_accounts/models.py lines 313-324
Impact: TWO CHECKS CAN HAVE THE SAME NUMBER
Issue: get_next_check_number() locks correctly, but is called OUTSIDE the atomic
       transaction in BankTransaction.save(). Simultaneous withdrawals could get
       duplicate check numbers.

================================================================================
HIGH SEVERITY BUGS
================================================================================

BUG #15: MULTIPLE CASE DEPOSITS - UPDATE LOGIC BROKEN
Location: /backend/apps/clients/models.py lines 386-405
Impact: Case balance INCORRECT when deposit amounts change
Issue: _update_case_deposit() uses .first() which may select wrong transaction
       when multiple deposits exist (e.g., after void/reissue). Case balance
       could be off by thousands of dollars.

BUG #7: SMART DELETE LOGIC HAS GAP
Location: /backend/apps/clients/api/views.py lines 266-313
Impact: Client with balance can be deleted (soft delete when shouldn't)
Issue: Balance calculation uses direct client FK only, missing transactions
       linked through case.client. Transactions with client=None, case=X
       won't be counted in balance check.

================================================================================
MEDIUM SEVERITY BUGS (API & CALCULATION)
================================================================================

BUG #2: VOIDED TRANSACTION AMOUNT INCONSISTENCY
        Amounts set to 0 but audit trail can't recover original amounts

BUG #3: TRANSFER_OUT AGGREGATION MISMATCH
        Grouped as withdrawal in Client.get_current_balance() but as separate
        TRANSFER in API summary (which doesn't exist in model!)

BUG #4: RECONCILED STATUS NOT EXPLICIT
        Status added to model but not explicitly included in balance calcs

BUG #8: TRANSACTION ORDERING INCONSISTENT
        balance_history shows oldest-first, transactions show newest-first
        Contradicts requirement: ALL displays must show oldest-first

BUG #9: SETTLEMENT DISTRIBUTION OVERFLOW
        No validation that distribution amounts don't exceed settlement total

BUG #10: UNMATCHED STATUS DOESN'T EXIST
         Filters by status='UNMATCHED' which never matches anything

BUG #11: AMOUNT RESPONSE FORMAT INCONSISTENT
         Some endpoints return Decimal, others return string
         Frontend can't handle mixed formats

BUG #12: VALIDATION CONFLICTS WITH EDIT RULES
         Requires ALL fields for PATCH, but only description editable on
         cleared transactions

BUG #13: ORPHANED TRANSACTIONS AFTER CASE DELETE
         on_delete=PROTECT prevents deletion but doesn't warn about orphans

BUG #14: IMPORT BATCH TRACKING INCOMPLETE
         Auto-created deposits don't track their import batch

================================================================================
LOW SEVERITY BUGS

BUG #16: ZERO BALANCE FLOAT COMPARISON
         Uses `== 0` which fails for rounding errors like 0.00000001
